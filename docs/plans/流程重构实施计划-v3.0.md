### v3.0 变更摘要

| 变更项 | 原设计 | 新设计 | 原因 |
|--------|--------|--------|------|
| **指派机制** | 逐任务手动指派（§2.8） | 批次创建时预指派生产员+质检员，任务自动继承；异常时允许转交 | 减少主管重复操作，一个批次由固定人员从头负责 |
| **CoA 流程** | QC_PASS→COA_PENDING→COA_SUBMITTED→COA_APPROVED→RELEASED（4步） | QC合格→系统自动生成CoA→QC预览提交→主管批准=自动放行（2步人工操作） | 减少无意义步骤，同角色多次操作无实际增益 |
| **QC 角色分离** | QC 操作允许 QC + OPERATOR | 严格限制仅 QC 角色可执行质检 | 符合 GMP 生产/质检分离原则 |
| **四眼原则** | 任务级校验 | 批次级预指派时校验（生产员 ≠ 质检员） | 在源头避免冲突，而非事后拦截 |
| **服务项目 HANDOVER** | IN_PRODUCTION↔HANDOVER 循环 | 去除 HANDOVER 状态，人员交接通过转交机制 | 状态机循环是反模式，转交机制更清晰 |
| **服务项目终止** | 统一使用 SCRAPPED | 引入"终止"概念（可分类原因），与服务项目特性适配 | 服务项目终止原因多样，不仅仅是报废 |
| **分化步骤** | 无 | 新增"分化诱导"阶段型步骤（可多轮记录，类似扩增培养） | 分化是一个阶段而非单一操作 |
| **生产过程采样** | 无 | 生产任务表单内嵌采样记录，关联 QC 待检记录 | 采样发生在生产过程中，不是独立流程步骤 |
| **批次级人员字段** | 无 | Batch 表新增 productionOperatorId / qcOperatorId | 支持预指派机制 |

## 1. 重构后的完整业务流程

### 1.1 细胞产品 / 试剂盒（简化后）

```
┌─────────────────────────────────────────────────────────────────────────┐
│  NEW ──→ IN_PRODUCTION ──→ QC_PENDING ──→ QC_IN_PROGRESS ──→ QC_PASS  │
│  (新建)    (生产中)         (待质检)       (质检中)         (质检合格)  │
│                                                                       │
│   ADMIN/         OPERATOR/      QC            QC              SYSTEM   │
│   SUPERVISOR     SUPERVISOR     (严格)        (严格)          (自动)    │
│                                                                       │
│                                                                      │
│  QC_PASS ──→ COA_SUBMITTED ──→ RELEASED                              │
│  (质检合格)   (CoA已提交)       (已放行)                                │
│                 │                 ↑                                     │
│           QC 预览并提交      主管批准                                   │
│           (自动生成CoA)     =自动放行                                    │
│                                                                       │
│  任何中间状态 ──→ SCRAPPED（批次级报废，强制填写原因）                    │
└─────────────────────────────────────────────────────────────────────────┘
```

**状态数**：8 个（NEW / IN_PRODUCTION / QC_PENDING / QC_IN_PROGRESS / QC_PASS / COA_SUBMITTED / RELEASED / SCRAPPED）

**废弃的状态**：
- `COA_PENDING`（质检合格后自动生成CoA草稿并直接进入待提交状态）
- `COA_APPROVED`（批准 = 放行，合并为一步）

**试剂盒额外状态**：
- `MATERIAL_PREP`（物料准备中）：NEW → MATERIAL_PREP → IN_PRODUCTION → ...
- 试剂盒状态数：9 个

### 1.2 服务项目（重构后）

```
┌──────────────────────────────────────────────────────────────────────────┐
│                                                                          │
│  NEW ──→ SAMPLE_RECEIVED ──→ IN_PRODUCTION ──→ IDENTIFICATION           │
│  (新建)    (样本已接收)       (生产中)         (鉴定中)                   │
│                                                                          │
│   OPERATOR  OPERATOR/          OPERATOR/        OPERATOR/                │
│             SUPERVISOR         SUPERVISOR       SUPERVISOR               │
│                                                                          │
│  IDENTIFICATION ──→ REPORT_PENDING ──→ COA_SUBMITTED ──→ RELEASED        │
│  (鉴定中)          (待生成报告)      (CoA已提交)       (已放行)           │
│                         ↑                 ↑                ↑             │
│                    鉴定完成          提交报告+CoA      主管批准            │
│                                      (自动生成CoA)   =自动放行            │
│                                                                          │
│  ┌── 异常路径 ──────────────────────────────────────────────────────┐    │
│  │ IN_PRODUCTION ──→ TERMINATED（终止：客户取消/样本问题/服务失败）   │    │
│  │ IDENTIFICATION ──→ IN_PRODUCTION（返工：鉴定失败，重新生产）        │    │
│  │ 任何状态 ──→ SCRAPPED（批次级报废，强制填写原因）                   │    │
│  └──────────────────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────────────────┘
```

**状态数**：9 个（NEW / SAMPLE_RECEIVED / IN_PRODUCTION / IDENTIFICATION / REPORT_PENDING / COA_SUBMITTED / RELEASED / TERMINATED / SCRAPPED）

**废弃的状态**：
- `HANDOVER`（人员交接通过转交机制替代）
- `COA_APPROVED`（批准 = 放行）

**新增的状态**：
- `TERMINATED`（终止）：服务项目特有的"非报废终止"，需要分类原因

### 1.3 服务项目的实际运作复杂性

服务项目不同于细胞产品/试剂盒的**线性流程**，存在以下特征：

```
【重编程建系 — 实际流程】
接收样本 → 样本处理 → 重编程操作(R1)
                              ├── 成功 → 克隆挑取 → 扩增培养 → 冻存（中途可暂停）
                              └── 失败 → 重编程操作(R2) → ...
                                              
鉴定（与生产并行进行）：
  QC 同时完成种子扩增、采样、鉴定实验
  鉴定实验失败 → 分析原因 → 重新进行鉴定
  全部通过 → 生成报告 → 提交审核 → 放行
```

**系统设计策略**：
- **批次状态机保持简化**（上面的6个主状态 + 2个异常状态）
- **复杂度下沉到任务级别**：同一 taskCode 可以出现多条记录（如 REPROGRAM-R1、REPROGRAM-R2）
- **任务支持"重做"**：标记原任务为 SKIPPED/FAILED，创建同类型新任务
- **QC并行工作**：QC 记录通过 `taskId` 关联到具体的生产任务，可在生产进行中创建
- **中途冻存**：通过生产任务表单记录冻存信息（冻存编号、冻存位置等），不影响批次状态

## 2. 关键设计决策（本次讨论确认）

### 2.1 报废机制

| 决策项 | 结论 |
|--------|------|
| 报废粒度 | **批次级报废**（整个批次终止，数据归档） |
| 是否强制原因 | ✅ 是，报废时必须填写原因（reason 字段必填） |
| 任务级报废 | ❌ 暂不实现（后续可扩展） |
| 后续对接 | 偏差分析模块（Phase 5+），报废记录可关联到偏差调查 |

### 2.2 指派机制

| 决策项 | 结论 |
|--------|------|
| 指派时机 | **批次启动时预指派**（创建批次或开始生产时一次性指派） |
| 指派内容 | 生产操作员（productionOperatorId）+ 质检员（qcOperatorId） |
| 任务继承 | 新创建的任务自动继承批次的预指派人员 |
| 四眼原则 | **预指派时校验**：生产员 ≠ 质检员 |
| 人员变更 | 操作员/质检员可发起"申请转交" → 主管在批次首页重新指定 → 更新所有未完成任务 |
| 主管指派 | SUPERVISOR/ADMIN 在批次首页统一管理指派 |

**数据模型变更**：

```prisma
model Batch {
  // ... 现有字段 ...

  // 新增：预指派人员
  productionOperatorId  String?
  productionOperatorName String?
  qcOperatorId          String?
  qcOperatorName        String?

  // 关联
  productionOperator    User?  @relation("BatchProductionOperator", ...)
  qcOperator            User?  @relation("BatchQcOperator", ...)
}
```

### 2.3 CoA 流程简化

| 决策项 | 结论 |
|--------|------|
| CoA 生成 | QC_PASS 后系统**自动生成** CoA 草稿 |
| CoA 提交 | **QC 角色**预览后点击提交（QC_PASS → COA_SUBMITTED） |
| CoA 审核 | SUPERVISOR 或 QA 批准 |
| 批准与放行 | **合并**：批准 = 自动放行（设置 actualEndDate） |
| 审计追踪 | 审计日志明确记录"因CoA批准而自动放行" |
| 后续扩展 | 客户交付、寄送等记录可追加到已放行批次的记录中（Phase 5+） |

**简化前 vs 简化后**：

```
简化前（4步人工）：
  QC_PASS → COA_PENDING → COA_SUBMITTED → COA_APPROVED → RELEASED
             (自动生成)    (OPERATOR提交)   (SUPERVISOR批准) (SUPERVISOR放行)

简化后（2步人工）：
  QC_PASS → COA_SUBMITTED → RELEASED
             (QC预览提交)  (主管批准=自动放行)
```

### 2.4 QC 角色严格分离

| 决策项 | 结论 |
|--------|------|
| 质检操作限制 | **仅 QC 角色**可执行质检（开始质检、质检合格/不合格、提交CoA） |
| 同人双角色 | 允许同一用户有 OPERATOR + QC 两个全局角色 |
| 产品权限独立 | 同一用户在不同产品上可以有不同的操作角色 |
| 四眼原则 | **同批次**的操作员和质检员**不能是同一人** |

**状态机变更**：

```typescript
// 修改前
QC_PENDING:    [{ roles: ['QC', 'OPERATOR'], ... }]
QC_IN_PROGRESS: [{ roles: ['QC', 'OPERATOR'], ... }]
COA_PENDING:    [{ roles: ['OPERATOR', 'SUPERVISOR'], ... }]

// 修改后
QC_PENDING:    [{ roles: ['QC'], ... }]
QC_IN_PROGRESS: [{ roles: ['QC'], ... }]
// QC_PASS → COA_SUBMITTED: 由 QC 提交
COA_SUBMITTED: [{ roles: ['QC'], action: 'submit_coa', ... }]  // QC预览并提交
COA_SUBMITTED: [{ roles: ['SUPERVISOR', 'QA'], action: 'approve', ... }]  // 批准=放行
```

### 2.5 服务项目终止机制

| 决策项 | 结论 |
|--------|------|
| 终止状态 | 新增 `TERMINATED`（与 `SCRAPPED` 并列的终态） |
| 终止 vs 报废 | `TERMINATED` = 服务未完成但非质量问题；`SCRAPPED` = 质量/生产问题 |
| 终止原因分类 | 客户取消 / 样本问题 / 需求变更 / 其他 |
| 是否强制原因 | ✅ 是 |
| 数据保留 | 所有已完成的生产记录和鉴定记录保留，供追溯 |

**TERMINATED 原因枚举**：

```typescript
const TERMINATION_REASONS = [
  'CUSTOMER_CANCEL',     // 客户取消
  'SAMPLE_ISSUE',        // 样本问题
  'REQUIREMENT_CHANGE',  // 需求变更
  'SERVICE_FAILURE',     // 服务失败（如重编程两轮均失败）
  'OTHER',               // 其他
]
```

### 2.6 去除 HANDOVER

| 决策项 | 结论 |
|--------|------|
| HANDOVER 状态 | **去除**，不再存在于状态机中 |
| 替代机制 | 人员转交（批次级预指派变更） |
| 转交流程 | 被指派人发起"申请转交" → 主管在批次首页重新指定 → 更新未完成任务 |

### 3.2 生产过程采样记录（方案 A+C）

**设计思路**：采样是生产过程中的一个动作，记录在对应的生产任务表单中。

```
┌─────────────────────────────────────────────────┐
│  扩增培养 P3                                     │
│                                                  │
│  ── 生产数据 ──                                  │
│  皿号: T03   培养基: mTeSR  传代比例: 1:6  ...   │
│                                                  │
│  ── 采样记录（可展开添加）──                      │
│  ✅ 支原体检测                                     │
│     样本编号: QC-260413-001                       │
│     取样时间: 2026-04-13 14:30                    │
│     取样人: 张三                                   │
│     检测项: 支原体                                 │
│     → 已生成 QC 待检记录                          │
│                                                  │
│  ✅ 核型分析                                       │
│     样本编号: ID-260413-001                       │
│     取样时间: 2026-04-13 14:35                    │
│     ...                                           │
│                                                  │
│  [+ 添加采样记录]                                 │
└─────────────────────────────────────────────────┘
```

**数据流**：

```
1. 操作员在生产任务表单中填写采样信息
2. 系统自动创建 QcRecord：
   - batchId: 当前批次
   - taskId: 当前生产任务
   - qcType: 'IN_PROCESS'  （区别于终检的 'ROUTINE'）
   - overallJudgment: 'PENDING'
   - testResults: 包含采样信息的 JSON
3. QC 人员在 QC 界面看到待处理样本
4. QC 录入检测结果 → 综合判定

批次最终质检时：
- 汇总所有 QC 记录（过程采样 + 终检）
- 形成最终报告 → CoA
```

### 3.3 任务重做机制（服务项目）

服务项目中某些步骤可以重做（如重编程第一轮失败）：

```
原任务：REPROGRAM (R1) → 状态设为 FAILED
新任务：REPROGRAM (R2) → 自动创建，sequenceNo + 0.1

任务列表：
├── 样本处理          ✅ COMPLETED
├── 重编程操作(R1)    ❌ FAILED      ← stepGroup: "REPROGRAM-R1"
├── 重编程操作(R2)    ⏳ PENDING      ← stepGroup: "REPROGRAM-R2"
├── 克隆挑取          🔒 LOCKED      ← 前序任务未完成
└── ...
```

**实现策略**：
- 通过 `stepGroup` 字段区分不同轮次（如 "REPROGRAM-R1"、"REPROGRAM-R2"）
- 原任务标记为 FAILED（而非删除，保留追溯）
- 新任务自动创建，sequenceNo = 原任务 sequenceNo + 0.1（保证排序正确）
- 后续任务在前序任务完成前锁定（不可开始）

### 3.4 阶段型任务

EXPANSION 和 DIFFERENTIATION 是阶段型任务，和 EBPR 的现有逻辑一致：
- 可以有多次记录（每次传代/操作一条）
- 每条记录通过 `stepGroup` 关联
- 新记录通过前端表单动态创建
- 完成判断：至少一条 COMPLETED 记录即视为该步骤完成

## 6. 风险与待确认项

| # | 风险/待确认 | 影响 | 建议 |
|---|------------|------|------|
| 1 | 服务项目的中途冻存是否需要批次状态变化？ | 可能需要 FROZEN 暂停状态 | 暂通过任务表单记录冻存信息，不改批次状态；如需暂停功能后续再加 |
| 2 | 鉴定失败重做的粒度？ | 是重做全部鉴定项还是单项目？ | 建议单项目重做（QcRecord 级别），不影响批次状态 |
| 3 | 客户需求变更如何处理？ | 可能影响鉴定要求、产品规格等 | 暂通过批次备注记录，后续可扩展"变更单"功能 |
| 4 | 不同产品的鉴定要求差异大 | 鉴定任务模板需要灵活配置 | 复用现有的 `identificationRequirements` JSON 字段，按产品配置 |
| 5 | 废弃状态的迁移 | 已有数据中 COA_PENDING / HANDOVER / COA_APPROVED 怎么处理？ | 保留枚举映射，前端展示为最新状态的等价状态；不改历史数据 |
