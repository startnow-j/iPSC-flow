## 1. 背景与目标

### 1.1 现状

MVP v2.0 已完成，系统当前支持：
- iPSC 细胞产品线（扩增冻存流程）的完整闭环
- 单一产品（IPSC-WT-001），4种全局角色
- 11种批次状态、16条转换规则的硬编码状态机
- eBPR 三步流程（种子复苏→扩增培养→收获冻存）
- 质检录入、CoA 自动生成与审核放行
- Dashboard、用户管理、审计日志

### 1.2 核心差距

| # | 差距 | 影响 | 对应阶段 |
|---|------|------|---------|
| 1 | 无产品线概念，系统仅支持单一产品 | 无法管理服务项目、试剂盒等不同业务流 | Phase 1 |
| 2 | 角色全局化，无产品线维度 | 无法实现"员工A是X产品的生产、Y产品的质检" | Phase 1 |
| 3 | 状态机硬编码单套流程 | 其他产品线有完全不同的状态流转 | Phase 2 |
| 4 | eBPR 表单固定3步 iPSC 流程 | 服务项目需要8+步骤，试剂盒需要不同的生产表单 | Phase 3 |
| 5 | 质检模板仅支持 iPSC 3项 | 不同产品线的质检项目完全不同 | Phase 3 |
| 6 | 无步骤级执行人指定 | 主管无法将具体步骤分配给特定人员 | Phase 1 |

## 2. 关键设计决策

### 2.1 用户-产品关联粒度

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| ~~A: 用户+产品线+角色~~ | 如"张三在细胞产品线是操作员" | 粒度粗，扩展性差 |
| **B: 用户+具体产品+角色** ✅ | 如"张三在IPSC-WT-001上是操作员，在NPC-001上是质检" | 粒度细，扩展性强 |

**决策**：选择方案 B（用户+具体产品+角色）

**理由**（用户确认）：
1. 更好的扩展性：未来产品增多时无需重构
2. 更符合实际业务：团队扩大后分工更专业，同一产品线的不同产品可能由不同人负责
3. 实现复杂度可接受：UserProductRole 关联表已在 schema 中定义

### 2.2 产品线与种子产品定义

| 产品线 | 枚举值 | 种子产品（Phase 1 创建） |
|--------|--------|------------------------|
| 服务项目 | `SERVICE` | 重编程建系服务(SRV-REPG-001)、基因编辑服务(SRV-EDIT-001) |
| 细胞产品 | `CELL_PRODUCT` | iPSC细胞株-野生型(IPSC-WT-001)【已有】、NPC神经前体细胞(NPC-001) |
| 试剂盒 | `KIT` | 神经分化试剂盒(KIT-ND-001)、iPSC维持培养基(KIT-MED-001) |

### 2.3 状态机参数化策略

| 产品线 | 状态流程 | 核心差异 |
|--------|---------|---------|
| 细胞产品 | NEW→IN_PRODUCTION→QC→CoA→RELEASED | 当前已实现，提取为模板 |
| 服务项目 | NEW→SAMPLE→IN_PRODUCTION→IDENTIFY→REPORT→DELIVERED | 含交接机制、鉴定融入生产、实验报告+CoA |
| 试剂盒 | NEW→PREP→IN_PRODUCTION→QC→CoA→RELEASED | 含物料准备阶段、效力验证(长周期) |

**策略**：3套流程模板常量，根据 `batch.productLine` 查找对应转换规则

### 2.4 批号格式

| 产品线 | 格式 | 示例 |
|--------|------|------|
| 服务项目 | `SRV-YYMMDD-NNN` | SRV-260618-001 |
| 细胞产品 | `{产品代码}-YYMMDD-NNN` | IPSC-260618-001 |
| 试剂盒 | `KIT-YYMMDD-NNN` | KIT-260618-001 |

### 2.5 AI 对话模式时机

**决策**：AI 对话模式放在 Phase 4（最后实施）

**理由**：
1. AI 模式需要依赖产品线上下文和流程模板才能正确工作
2. 前三个阶段完成后，AI 才有完整的"可操作空间"
3. 表单模式先跑通所有产品线流程，AI 模式作为增强叠加

## 4. Phase 2 详细设计：多产品线状态机

### 4.1 目标

- 3套流程模板常量（按产品线分组）
- 参数化状态机：`getTransitions(productLine, status)` 替代硬编码 `BATCH_TRANSITIONS`
- BatchStatus 枚举扩展（一次性新增全部）
- 去掉 REJECTED 批次状态（CoA 表内保持 DRAFT/SUBMITTED/REJECTED）
- 任务模板骨架定义（Phase 3 实现表单 UI）
- 各产品线的状态流转全流程跑通

### 4.2 用户确认的设计决策（v1.1 更新）

#### 4.2.1 REJECTED 状态处理

**决策**：去掉 REJECTED 作为独立批次状态

| 场景 | 原方案 | 新方案 |
|------|--------|--------|
| 细胞产品 CoA 退回 | COA_SUBMITTED → REJECTED → COA_SUBMITTED | COA_SUBMITTED 状态下退回（CoA 表 status→DRAFT），批次状态不变 |
| 服务项目报告退回 | COA_SUBMITTED → REPORT_PENDING | COA_SUBMITTED → REPORT_PENDING（批次状态直接回退） |

**实现**：
- BatchStatus 枚举删除 REJECTED
- CoaStatus 枚举保持 DRAFT/SUBMITTED/REJECTED/APPROVED
- 细胞产品的 CoA 退回：只更新 CoA 表 status 为 DRAFT，批次保持 COA_SUBMITTED
- 服务项目的报告退回：批次状态 COA_SUBMITTED → REPORT_PENDING

#### 4.2.2 服务项目鉴定（IDENTIFICATION）机制

**决策**：IDENTIFICATION 作为批次状态（宏观阶段），内部子步骤通过 ProductionTask 管理（微观执行）

- 批次进入 IDENTIFICATION = 标志"重编程完成，进入鉴定阶段"
- 批次离开 IDENTIFICATION = 标志"所有鉴定项目完成"
- 鉴定子步骤（整合检测、核型、多能性、STR 等）都是 ProductionTask
- 鉴定任务没有严格 sequenceNo 排序，可分配给不同人员
- 鉴定完成由操作员/主管手动触发（因判定"全部完成"的逻辑灵活）
- 基因编辑的两轮克隆挑选：IDENTIFICATION → IN_PRODUCTION → IDENTIFICATION（状态机回退）

#### 4.2.3 交接机制（HANDOVER）

**决策**：允许多次交接，HANDOVER 作为轻量过渡状态

```
IN_PRODUCTION ⇄ HANDOVER（可循环多次）
```

- 每次交接创建一个 HANDOVER 类型的 ProductionTask
- 记录交接方、接收方、交接说明
- 接收确认后才能进入下一个状态
- 不限制交接次数

#### 4.2.4 鉴定不通过处理

**决策**：由主管判断如何处理

```
IDENTIFICATION
├── REPORT_PENDING    → 鉴定全部通过
├── IN_PRODUCTION     → 返工（主管决定）
└── SCRAPPED          → 报废（主管决定）
```

- 多能性质控不通过 → 可能只是实验问题 → IDENTIFICATION → IN_PRODUCTION
- 核型异常 → 该克隆不可救 → IDENTIFICATION → SCRAPPED
- 基因型不纯 → 换子克隆 → IDENTIFICATION → IN_PRODUCTION

#### 4.2.5 服务项目报告+CoA 流程

**决策**：简化流程，跳过独立 QC 环节

```
REPORT_PENDING → COA_SUBMITTED → RELEASED
                     ↑
                     └── REPORT_PENDING（退回修改）
```

- 跳过 QC_PASS → COA_PENDING → COA_SUBMITTED 链路
- 到报告环节时理论上所有鉴定/质检已通过
- 批准后直接放行（RELEASED）

#### 4.2.6 试剂盒效力验证

**决策**：必须完成后才能放行

- 效力验证作为可选质检项，创建批次时配置
- 如果指令单要求做效力检测 → 必须等完成 → 才能放行
- 效力检测 15-30 天的等待是正常业务过程（QC_IN_PROGRESS 状态持续）

#### 4.2.7 状态机模板结构

**决策**：方案A — 按产品线分组

```typescript
const TRANSITION_TEMPLATES: Record<ProductLine, Record<string, TransitionRule[]>> {
  CELL_PRODUCT: { /* 独立完整流程 */ },
  SERVICE:       { /* 独立完整流程 */ },
  KIT:           { /* 独立完整流程 */ },
}
```

**理由**：三条产品线差异远大于共性，分组方案可读性、维护性、安全性均优于扁平混合方案。

#### 4.2.8 任务模板策略

**决策**：Phase 2 定义任务骨架 + Phase 3 实现表单

- Phase 2：定义每个状态转换时创建哪些任务（taskCode/taskName/sequenceNo）
- Phase 3：为每种任务类型开发具体表单组件
- 鉴定要求在创建批次时由主管勾选，存入 Batch.identificationRequirements（JSON）
- 主管后续可追加/修改鉴定要求（动态创建新鉴定任务）

### 4.5 三套完整流程模板

#### 4.5.1 细胞产品（CELL_PRODUCT）

```
NEW → IN_PRODUCTION → QC_PENDING → QC_IN_PROGRESS → QC_PASS → COA_PENDING(自动)
  → COA_SUBMITTED → COA_APPROVED → RELEASED
                                                             ↓
                                                     (CoA退回,CoA→DRAFT)
                                                             ↓
                                                     COA_SUBMITTED(批次不变)
                                                             ↓
                                                        COA_SUBMITTED → COA_APPROVED

QC_FAIL → QC_PENDING(返工) / SCRAPPED(报废)
任意非终态 → SCRAPPED(报废)
```

## 6. Phase 4 详细设计：AI 对话模式

### 6.1 目标

- 意图层（Intent Layer）实现
- LLM 集成（后端 z-ai-web-dev-sdk）
- 确认卡片（Confirmation Card Protocol）
- 上下文服务（Context Service）
- 对话工作台 UI

### 6.2 依赖关系

Phase 4 必须在 Phase 1-3 完成后实施：
- 意图层依赖完整的产品线和流程模板定义
- 上下文服务依赖所有产品线的业务数据
- 确认卡片依赖所有表单的字段定义和 ai_hint

### 6.3 核心模块

```
Phase 4 模块分解：
├── 后端
│   ├── 意图层 (Intent Layer)
│   │   ├── 意图定义（基于已有 PRD 中的 operation_intent 设计）
│   │   ├── 意图解析器（LLM 提取意图+参数）
│   │   └── 意图执行器（调用对应业务服务）
│   ├── 上下文服务 (Context Service)
│   │   ├── 批次上下文（当前状态、历史记录、下一步建议）
│   │   └── 默认值服务（培养基批号、操作人等自动填充）
│   ├── 确认卡片服务 (Card Protocol)
│   │   ├── 卡片构建器（根据意图和参数生成卡片数据）
│   │   └── 卡片渲染协议（前端通用渲染器）
│   └── 对话 API
│       ├── POST /api/conversation — 发送消息
│       ├── GET /api/conversation/[id] — 获取对话历史
│       └── POST /api/conversation/[id]/confirm — 确认卡片
│
└── 前端
    ├── 对话工作台组件
    │   ├── 聊天窗口（消息列表、输入框）
    │   ├── 确认卡片渲染器（通用组件）
    │   └── 模式切换（对话/表单）
    └── 对话 Store（Zustand）
```

## 8. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 状态机改造影响现有流程 | 细胞产品线当前可用的功能回退 | Phase 2 先确保现有流程不受影响，再逐步添加新模板 |
| 产品级权限检查过于复杂 | 开发周期延长 | Phase 1 向后兼容，渐进式改造 |
| 6+产品的表单开发量大 | Phase 3 周期过长 | 优先实现最关键的表单，次要表单用通用表单替代 |
| SQLite 不支持某些高级查询 | 多产品线联查性能问题 | 当前数据量小，SQLite 足够；后续可迁移 |
