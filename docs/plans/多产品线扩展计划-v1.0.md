## 1. 背景与目标

### 1.1 现状

MVP v2.0 已完成，系统当前支持：
- iPSC 细胞产品线（扩增冻存流程）的完整闭环
- 单一产品（IPSC-WT-001），4种全局角色
- 11种批次状态、16条转换规则的硬编码状态机
- eBPR 三步流程（种子复苏→扩增培养→收获冻存）
- 质检录入、CoA 自动生成与审核放行
- Dashboard、用户管理、审计日志

### 1.2 核心差距

| # | 差距 | 影响 | 对应阶段 |
|---|------|------|---------|
| 1 | 无产品线概念，系统仅支持单一产品 | 无法管理服务项目、试剂盒等不同业务流 | Phase 1 |
| 2 | 角色全局化，无产品线维度 | 无法实现"员工A是X产品的生产、Y产品的质检" | Phase 1 |
| 3 | 状态机硬编码单套流程 | 其他产品线有完全不同的状态流转 | Phase 2 |
| 4 | eBPR 表单固定3步 iPSC 流程 | 服务项目需要8+步骤，试剂盒需要不同的生产表单 | Phase 3 |
| 5 | 质检模板仅支持 iPSC 3项 | 不同产品线的质检项目完全不同 | Phase 3 |
| 6 | 无步骤级执行人指定 | 主管无法将具体步骤分配给特定人员 | Phase 1 |

### 1.4 验收标准

每阶段完成后必须达到：
- **3条产品线 × 各至少2个产品** 的全流程跑通
- 新建批次 → 生产记录 → 质检 → CoA → 放行的完整闭环

## 2. 关键设计决策

### 2.1 用户-产品关联粒度

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| ~~A: 用户+产品线+角色~~ | 如"张三在细胞产品线是操作员" | 粒度粗，扩展性差 |
| **B: 用户+具体产品+角色** ✅ | 如"张三在IPSC-WT-001上是操作员，在NPC-001上是质检" | 粒度细，扩展性强 |

**决策**：选择方案 B（用户+具体产品+角色）

**理由**（用户确认）：
1. 更好的扩展性：未来产品增多时无需重构
2. 更符合实际业务：团队扩大后分工更专业，同一产品线的不同产品可能由不同人负责
3. 实现复杂度可接受：UserProductRole 关联表已在 schema 中定义

### 2.2 产品线与种子产品定义

| 产品线 | 枚举值 | 种子产品（Phase 1 创建） |
|--------|--------|------------------------|
| 服务项目 | `SERVICE` | 重编程建系服务(SRV-REPG-001)、基因编辑服务(SRV-EDIT-001) |
| 细胞产品 | `CELL_PRODUCT` | iPSC细胞株-野生型(IPSC-WT-001)【已有】、NPC神经前体细胞(NPC-001) |
| 试剂盒 | `KIT` | 神经分化试剂盒(KIT-ND-001)、iPSC维持培养基(KIT-MED-001) |

### 2.3 状态机参数化策略

| 产品线 | 状态流程 | 核心差异 |
|--------|---------|---------|
| 细胞产品 | NEW→IN_PRODUCTION→QC→CoA→RELEASED | 当前已实现，提取为模板 |
| 服务项目 | NEW→SAMPLE→IN_PRODUCTION→IDENTIFY→REPORT→DELIVERED | 含交接机制、鉴定融入生产、实验报告+CoA |
| 试剂盒 | NEW→PREP→IN_PRODUCTION→QC→CoA→RELEASED | 含物料准备阶段、效力验证(长周期) |

**策略**：3套流程模板常量，根据 `batch.productLine` 查找对应转换规则

### 2.4 批号格式

| 产品线 | 格式 | 示例 |
|--------|------|------|
| 服务项目 | `SRV-YYMMDD-NNN` | SRV-260618-001 |
| 细胞产品 | `{产品代码}-YYMMDD-NNN` | IPSC-260618-001 |
| 试剂盒 | `KIT-YYMMDD-NNN` | KIT-260618-001 |

### 2.5 AI 对话模式时机

**决策**：AI 对话模式放在 Phase 4（最后实施）

**理由**：
1. AI 模式需要依赖产品线上下文和流程模板才能正确工作
2. 前三个阶段完成后，AI 才有完整的"可操作空间"
3. 表单模式先跑通所有产品线流程，AI 模式作为增强叠加

#### 3.3.2 权限改造

当前权限检查使用全局 `roles` 数组（any-match）。Phase 1 需增加产品线维度：

```typescript
// 当前逻辑（全局角色）
if (!hasAnyRole(roles, ['SUPERVISOR', 'OPERATOR'])) return 403

// Phase 1 目标（产品级角色）
// 1. 获取用户在该产品上的角色
const userProductRoles = await getUserProductRoles(userId, batch.productId)
// 2. 检查是否满足要求
if (!hasAnyRole(userProductRoles, requiredRoles)) return 403
```

**过渡策略**：
- Phase 1 保持向后兼容：优先检查产品级角色，回退到全局角色
- 管理员（ADMIN）始终拥有所有权限

### 4.1 目标

### 4.2 状态机模板设计

```typescript
// src/lib/services/state-machine.ts 改造

// 状态转换规则按产品线分组
const TRANSITION_TEMPLATES: Record<ProductLine, Record<string, TransitionRule[]>> = {
  CELL_PRODUCT: {
    // 当前已有的 iPSC 扩增冻存流程（直接迁移）
    NEW: [...],
    IN_PRODUCTION: [...],
    // ...
  },
  SERVICE: {
    NEW: [
      { to: 'SAMPLE_RECEIVED', action: 'receive_sample', roles: ['OPERATOR'], label: '接收样本' },
    ],
    SAMPLE_RECEIVED: [
      { to: 'IN_PRODUCTION', action: 'start_production', roles: ['OPERATOR', 'SUPERVISOR'], label: '开始生产' },
    ],
    IN_PRODUCTION: [
      { to: 'HANDOVER', action: 'request_handover', roles: ['OPERATOR'], label: '申请交接' },
      { to: 'IDENTIFICATION', action: 'start_identification', roles: ['OPERATOR'], label: '开始鉴定' },
    ],
    HANDOVER: [
      { to: 'IN_PRODUCTION', action: 'accept_handover', roles: ['OPERATOR'], label: '接收交接' },
    ],
    IDENTIFICATION: [
      { to: 'REPORT_PENDING', action: 'complete_identification', roles: ['OPERATOR'], label: '完成鉴定' },
    ],
    REPORT_PENDING: [
      { to: 'COA_SUBMITTED', action: 'submit_report', roles: ['SUPERVISOR'], label: '提交报告+CoA' },
    ],
    COA_SUBMITTED: [
      { to: 'RELEASED', action: 'approve', roles: ['SUPERVISOR', 'QA'], label: '批准交付' },
      { to: 'REPORT_PENDING', action: 'reject', roles: ['SUPERVISOR', 'QA'], label: '退回修改' },
    ],
    // ...
  },
  KIT: {
    NEW: [
      { to: 'MATERIAL_PREP', action: 'start_material_prep', roles: ['OPERATOR'], label: '开始备料' },
    ],
    MATERIAL_PREP: [
      { to: 'IN_PRODUCTION', action: 'start_production', roles: ['OPERATOR'], label: '开始配制' },
    ],
    IN_PRODUCTION: [
      { to: 'QC_PENDING', action: 'complete_production', roles: ['OPERATOR'], label: '提交质检' },
    ],
    // ... 质检和CoA流程与细胞产品类似
  },

// 向后兼容的查询函数
export function getTransitions(productLine: ProductLine): Record<string, TransitionRule[]> {
  return TRANSITION_TEMPLATES[productLine] ?? TRANSITION_TEMPLATES.CELL_PRODUCT
```

### 5.3 质检模板差异化

## 6. Phase 4 详细设计：AI 对话模式

### 6.1 目标

- 意图层（Intent Layer）实现
- LLM 集成（后端 z-ai-web-dev-sdk）
- 确认卡片（Confirmation Card Protocol）
- 上下文服务（Context Service）
- 对话工作台 UI

### 6.2 依赖关系

Phase 4 必须在 Phase 1-3 完成后实施：
- 意图层依赖完整的产品线和流程模板定义
- 上下文服务依赖所有产品线的业务数据
- 确认卡片依赖所有表单的字段定义和 ai_hint

### 6.3 核心模块

```
Phase 4 模块分解：
├── 后端
│   ├── 意图层 (Intent Layer)
│   │   ├── 意图定义（基于已有 PRD 中的 operation_intent 设计）
│   │   ├── 意图解析器（LLM 提取意图+参数）
│   │   └── 意图执行器（调用对应业务服务）
│   ├── 上下文服务 (Context Service)
│   │   ├── 批次上下文（当前状态、历史记录、下一步建议）
│   │   └── 默认值服务（培养基批号、操作人等自动填充）
│   ├── 确认卡片服务 (Card Protocol)
│   │   ├── 卡片构建器（根据意图和参数生成卡片数据）
│   │   └── 卡片渲染协议（前端通用渲染器）
│   └── 对话 API
│       ├── POST /api/conversation — 发送消息
│       ├── GET /api/conversation/[id] — 获取对话历史
│       └── POST /api/conversation/[id]/confirm — 确认卡片
│
└── 前端
    ├── 对话工作台组件
    │   ├── 聊天窗口（消息列表、输入框）
    │   ├── 确认卡片渲染器（通用组件）
    │   └── 模式切换（对话/表单）
    └── 对话 Store（Zustand）
```

### 6.4 Phase 4 验收清单

- [ ] 对话工作台 UI 可用（消息列表、输入框、确认卡片）
- [ ] 能通过自然语言完成 iPSC 传代记录录入
- [ ] AI 提取数据→确认卡片→用户确认→写入数据库 完整流程
- [ ] 上下文感知（AI 知道当前批次状态和历史）
- [ ] 表单/对话模式无缝切换
- [ ] 审计日志区分 FORM_SUBMIT / AI_CONVERSATION

## 8. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 状态机改造影响现有流程 | 细胞产品线当前可用的功能回退 | Phase 2 先确保现有流程不受影响，再逐步添加新模板 |
| 产品级权限检查过于复杂 | 开发周期延长 | Phase 1 向后兼容，渐进式改造 |
| 6+产品的表单开发量大 | Phase 3 周期过长 | 优先实现最关键的表单，次要表单用通用表单替代 |
| SQLite 不支持某些高级查询 | 多产品线联查性能问题 | 当前数据量小，SQLite 足够；后续可迁移 |
