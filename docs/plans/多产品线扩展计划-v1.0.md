## 1. 背景与目标

### 1.1 现状

MVP v2.0 已完成，系统当前支持：
- iPSC 细胞产品线（扩增冻存流程）的完整闭环
- 单一产品（IPSC-WT-001），4种全局角色
- 11种批次状态、16条转换规则的硬编码状态机
- eBPR 三步流程（种子复苏→扩增培养→收获冻存）
- 质检录入、CoA 自动生成与审核放行
- Dashboard、用户管理、审计日志

### 1.2 核心差距

| # | 差距 | 影响 | 对应阶段 |
|---|------|------|---------|
| 1 | 无产品线概念，系统仅支持单一产品 | 无法管理服务项目、试剂盒等不同业务流 | Phase 1 |
| 2 | 角色全局化，无产品线维度 | 无法实现"员工A是X产品的生产、Y产品的质检" | Phase 1 |
| 3 | 状态机硬编码单套流程 | 其他产品线有完全不同的状态流转 | Phase 2 |
| 4 | eBPR 表单固定3步 iPSC 流程 | 服务项目需要8+步骤，试剂盒需要不同的生产表单 | Phase 3 |
| 5 | 质检模板仅支持 iPSC 3项 | 不同产品线的质检项目完全不同 | Phase 3 |
| 6 | 无步骤级执行人指定 | 主管无法将具体步骤分配给特定人员 | Phase 1 |

## 2. 关键设计决策

### 2.1 用户-产品关联粒度

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| ~~A: 用户+产品线+角色~~ | 如"张三在细胞产品线是操作员" | 粒度粗，扩展性差 |
| **B: 用户+具体产品+角色** ✅ | 如"张三在IPSC-WT-001上是操作员，在NPC-001上是质检" | 粒度细，扩展性强 |

**决策**：选择方案 B（用户+具体产品+角色）

**理由**（用户确认）：
1. 更好的扩展性：未来产品增多时无需重构
2. 更符合实际业务：团队扩大后分工更专业，同一产品线的不同产品可能由不同人负责
3. 实现复杂度可接受：UserProductRole 关联表已在 schema 中定义

### 2.2 产品线与种子产品定义

| 产品线 | 枚举值 | 种子产品（Phase 1 创建） |
|--------|--------|------------------------|
| 服务项目 | `SERVICE` | 重编程建系服务(SRV-REPG-001)、基因编辑服务(SRV-EDIT-001) |
| 细胞产品 | `CELL_PRODUCT` | iPSC细胞株-野生型(IPSC-WT-001)【已有】、NPC神经前体细胞(NPC-001) |
| 试剂盒 | `KIT` | 神经分化试剂盒(KIT-ND-001)、iPSC维持培养基(KIT-MED-001) |

### 2.3 状态机参数化策略

| 产品线 | 状态流程 | 核心差异 |
|--------|---------|---------|
| 细胞产品 | NEW→IN_PRODUCTION→QC→CoA→RELEASED | 当前已实现，提取为模板 |
| 服务项目 | NEW→SAMPLE→IN_PRODUCTION→IDENTIFY→REPORT→DELIVERED | 含交接机制、鉴定融入生产、实验报告+CoA |
| 试剂盒 | NEW→PREP→IN_PRODUCTION→QC→CoA→RELEASED | 含物料准备阶段、效力验证(长周期) |

**策略**：3套流程模板常量，根据 `batch.productLine` 查找对应转换规则

### 2.5 AI 对话模式时机

**决策**：AI 对话模式放在 Phase 4（最后实施）

**理由**：
1. AI 模式需要依赖产品线上下文和流程模板才能正确工作

3. 表单模式先跑通所有产品线流程，AI 模式作为增强叠加

### 2.6 用户权限体系设计

> **文档版本**：v2.0
> **状态**：已确认（QA/QC 分离 + 产品线归属 + 四眼原则），待实施
> **关联数据模型**：User（全局角色）、UserProductLine（产品线归属）、UserProductRole（产品级操作权限）

#### 2.6.1 设计原则

**QA/QC 分离，两级配置，四眼原则**

权限体系基于以下核心原则：

1. **角色分离**：QA（质量保证）是管理类角色，QC（质量控制）是操作类角色，两者职责完全不同
2. **两级配置**：ADMIN 管账号 + 全局角色 + 产品线归属；SUPERVISOR 管自己产品线内的具体产品操作权限
3. **双轨验证**：管理类操作看职级（全局角色 + 产品线归属），操作类操作看培训（全局资质 + 产品授权）
4. **四眼原则**：同一个人在同一个批次中不能既执行生产任务又执行质检任务

#### 2.6.2 角色体系：五个角色，两类性质

| 角色 | 枚举值 | 性质 | 职责定位 | 粒度 |
|------|--------|------|---------|------|
| 管理员 | `ADMIN` | **管理类** | 系统管理、用户管理 | 全局，天然全通 |
| 生产主管 | `SUPERVISOR` | **管理类** | 批次创建、CoA 审核、任务分配、产品权限配置 | 产品线级，需指定 |
| 质量保证 | `QA` | **管理类** | 审核体系、批准 SOP、审批偏差、终审 QC 结果 | 产品线级，需指定 |
| 质量控制 | `QC` | **操作类** | 执行质检实验（核型、流式、活力、菌检等） | 产品级，需具体授权 |
| 生产操作员 | `OPERATOR` | **操作类** | 执行生产任务（培养、冻存、传代等） | 产品级，需具体授权 |

**主角色优先级**：`ADMIN > SUPERVISOR > QA > QC > OPERATOR`

**QA vs QC 的本质区别**：

| | QA（质量保证） | QC（质量控制） |
|--|---------------|---------------|
| 性质 | 管理/监督 | 操作/执行 |
| 做什么 | 审核体系、批准 SOP、审批偏差 | 做实验——菌检、核型、流式、活力 |
| 工作场景 | 办公桌、会议室 | 实验台 |
| 需要产品级培训？ | ❌ 审流程合规性 | ✅ 不同产品检测方法不同 |

- 每个用户可以拥有多个全局角色（`User.roles` JSON数组）
- 一个用户可以同时拥有管理类 + 操作类角色（如 `["SUPERVISOR", "QC"]`）

#### 2.6.3 三级权限数据模型

```
第①层：User.roles = ["OPERATOR","QC"]              ← ADMIN 配置
       回答："张三有哪种资质？"

第②层：UserProductLine                              ← ADMIN 配置
       userId → 张三, productLine → CELL_PRODUCT
       回答："张三属于哪条产品线？活动范围是什么？"

第③层：UserProductRole (userId+productId+roles)     ← SUPERVISOR 配置
       userId → 张三, productId → IPSC-WT-001, roles → ["OPERATOR"]
       回答："张三能碰哪个产品？做哪种活？"
```

**数据表**：

| 表 | 配置者 | 存储 |
|---|--------|------|
| `User.roles` | ADMIN | 全局角色 JSON数组：`["OPERATOR","QC"]` |
| `UserProductLine` | ADMIN | 产品线归属：`userId + productLine` |
| `UserProductRole` | SUPERVISOR | 产品操作权限：`userId + productId + roles`（只存操作类角色） |

**粒度规则**（用户已确认）：
- ✅ **生产操作和质检执行**：粒度到具体产品（实验人员技能培训和工作分配到产品）
- ✅ **生产管理和质量保证**：粒度到产品线（SUPERVISOR/QA 管整条线，不需要逐产品分配）
- ✅ **管理员**：天然全通，不需要任何产品线/产品级分配

#### 2.6.4 权限判断逻辑（双轨验证）

废弃旧的"合并取并集"逻辑，改为管理类和操作类两条独立的检查路径：

```
用户能否执行操作？
│
├─ 如果是管理类操作（创建批次、审核CoA、分配任务、审批偏差...）
│   └─ hasGlobalRole(requiredRoles)                → 是/否
│   └─ isAdmin OR hasProductLine(productLine)      → 是/否（ADMIN 天然通过）
│
└─ 如果是操作类操作（执行生产、执行质检...）
    ├─ isAdmin                                     → 是 → 直接通过
    ├─ hasGlobalRole(requiredRoles)                → 否 → 拒绝（无此资质）
    └─ hasProductRole(productId, requiredRoles)    → 否 → 拒绝（该产品未授权）
                                                     是 → 通过
```

**代码表达**：

```typescript
// 管理类操作：全局角色 + 产品线归属
function canManage(user: User, productLine: string, requiredRoles: string[]): boolean {
  return isAdmin(user.roles)
      || (hasAnyRole(user.roles, requiredRoles)
          && hasProductLine(user.productLines, productLine))
}

// 操作类操作：全局资质 + 产品授权
function canOperate(user: User, productId: string, requiredRoles: string[]): boolean {
  return isAdmin(user.roles)
      || (hasAnyRole(user.roles, requiredRoles)
          && hasProductRole(user.productRoles, productId, requiredRoles))
}
```

#### 2.6.5 操作权限矩阵

| 操作 | 类型 | 全局角色要求 | 范围要求 | 判断函数 |
|------|------|------------|---------|---------|
| 查看批次列表/详情 | 查看 | 任意已登录 | ❌ | — |
| **执行生产任务** | 操作 | `OPERATOR` | ✅ 该产品有 `OPERATOR` 产品角色 | `canOperate()` |
| **执行质检** | 操作 | `QC` | ✅ 该产品有 `QC` 产品角色 | `canOperate()` |
| 创建批次 | 管理 | `SUPERVISOR`/`ADMIN` | ✅ 属于该产品线 | `canManage()` |
| 审核 CoA | 管理 | `SUPERVISOR`/`QA`/`ADMIN` | ✅ 属于该产品线 | `canManage()` |
| 分配任务 | 管理 | `SUPERVISOR`/`ADMIN` | ✅ 属于该产品线 | `canManage()` |
| 审批偏差/变更 | 管理 | `QA`/`SUPERVISOR`/`ADMIN` | ✅ 属于该产品线 | `canManage()` |
| 管理用户账号 | 管理 | `ADMIN` | ❌ 全局 | `canManage()` |
| 管理产品定义 | 管理 | `ADMIN`/`SUPERVISOR` | ✅ 属于该产品线 | `canManage()` |

#### 2.6.6 四眼原则（同批次生产/质检隔离）

> **规则：即使一个人在某个产品上同时拥有 OPERATOR 和 QC 权限，也不允许在同一个批次中既执行生产任务又执行质检任务。**

**拦截时机**：SUPERVISOR（或 ADMIN）指派任务时

```
指派 QC 任务到批次 A：
  → 查询：该人是否在批次 A 上已有 PRODUCTION 类型的已提交/进行中任务？
     → 是 → ❌ 拒绝，"已执行该批次的生产任务，不可同时执行质检"
     → 否 → ✅ 允许

指派 PRODUCTION 任务到批次 A：
  → 查询：该人是否在批次 A 上已有 QC 类型的已提交/进行中任务？
     → 是 → ❌ 拒绝，"已执行该批次的质检任务，不可同时执行生产"
     → 否 → ✅ 允许
```

**边界情况**：
| 场景 | 结果 |
|------|------|
| 张三做了批次 A 的生产，做批次 B 的质检 | ✅ 允许（不同批次） |
| 张三做了批次 A 的生产，做批次 A 的质检 | ❌ 拒绝 |
| ADMIN 指派绕过 | ❌ 同样受四眼原则约束 |

**实现层级**：
- **后端**：任务指派 API（`ProductionTask` / `QcRecord`）校验
- **前端**：指派人选择器中对冲突人员标记警告

### 2.8 任务指派与双签机制

> **文档版本**：v1.0
> **状态**：已确认（流程测试阶段用户反馈），待实施
> **触发场景**：流程测试中发现 SUPERVISOR 启动生产后，没有指派操作员的界面

#### 2.8.1 问题分析

当前系统任务指派存在以下缺陷：

| # | 问题 | 当前行为 | 期望行为 |
|---|------|---------|--------|
| 1 | 隐式指派 | 谁点"开始生产"就自动成为第一个任务的操作人 | SUPERVISOR 启动生产后，任务应处于"待指派"状态 |
| 2 | 无指派 UI | 没有任何"指派操作员"的界面 | SUPERVISOR 可在批次详情页主动指派任务 |
| 3 | 单人操作 | 仅 `assigneeId` 一个操作人字段 | 关键环节需要双人（操作人 + 复核人） |
| 4 | 后续任务无人管 | 第一个任务之后的任务全部 `assigneeId: null` | SUPERVISOR 可提前指派所有任务的操作人 |

#### 2.8.2 设计方案：主操作人 + 复核人

为 ProductionTask 新增 `reviewerId` / `reviewerName` 字段：

| 角色 | 用途 | 触发时机 | 可否为空 |
|------|------|---------|--------|
| **主操作人** (`assigneeId`) | 执行任务操作 | SUPERVISOR 指派，或操作员自行开始任务 | 可为空（待指派） |
| **复核人** (`reviewerId`) | 核验确认任务结果 | SUPERVISOR 指派 | 可为空（仅关键步骤需要） |

**任务完成的双签流程**：

```
操作人执行任务 → 填写表单 → 提交（任务状态: COMPLETED）
  → 如果有复核人 → 复核人核验 → 签字确认（任务状态: REVIEWED）
  → 如果无复核人 → 直接完成
```

#### 2.8.3 数据模型变更

```prisma
model ProductionTask {
  // ... 现有字段 ...

  // 执行人（已有）
  assigneeId      String?
  assigneeName    String?

  // 复核人（新增）
  reviewerId      String?
  reviewerName    String?
  reviewedAt      DateTime?    // 复核完成时间

  // 状态扩展
  status          TaskStatus @default(PENDING)
  // TaskStatus 枚举新增 REVIEWED:
  // PENDING → IN_PROGRESS → COMPLETED → REVIEWED（有复核人时）
  // PENDING → IN_PROGRESS → COMPLETED（无复核人时）

  // Relations
  assignee        User?      @relation(fields: [assigneeId], references: [id])
  reviewer        User?      @relation("TaskReviewer", fields: [reviewerId], references: [id])
}
```

**TaskStatus 枚举新增**：

```prisma
enum TaskStatus {
  PENDING
  IN_PROGRESS
  COMPLETED
  REVIEWED        // 新增：复核人已确认
  SKIPPED
}
```

#### 2.8.4 任务生命周期（修改后）

```
SUPERVISOR 启动生产
  → 自动创建任务（assigneeId: null, reviewerId: null）
  → 所有任务处于 PENDING 状态（不再自动将第一个任务设为 IN_PROGRESS）

SUPERVISOR 指派任务（新增操作）
  → 设置 assigneeId（主操作人）
  → 可选设置 reviewerId（复核人）
  → 四眼原则校验（§2.6.6）

操作员开始任务
  → 任务状态 PENDING → IN_PROGRESS
  → 自动设置 actualStart

操作员完成任务
  → 填写表单数据
  → 任务状态 IN_PROGRESS → COMPLETED
  → 设置 actualEnd
  → 如果有 reviewerId → 通知复核人

复核人确认（新增操作）
  → 检查任务数据和表单
  → 任务状态 COMPLETED → REVIEWED
  → 设置 reviewedAt

SUPERVISOR 推进批次（状态转换）
  → 前序任务需为 COMPLETED 或 REVIEWED 状态
```

## 4. Phase 2 详细设计：多产品线状态机

### 4.1 目标

- 3套流程模板常量（按产品线分组）
- 参数化状态机：`getTransitions(productLine, status)` 替代硬编码 `BATCH_TRANSITIONS`

- 去掉 REJECTED 批次状态（CoA 表内保持 DRAFT/SUBMITTED/REJECTED）
- 任务模板骨架定义（Phase 3 实现表单 UI）
- 各产品线的状态流转全流程跑通

### 4.2 用户确认的设计决策（v1.1 更新）

#### 4.2.1 REJECTED 状态处理

**决策**：去掉 REJECTED 作为独立批次状态

| 场景 | 原方案 | 新方案 |
|------|--------|--------|
| 细胞产品 CoA 退回 | COA_SUBMITTED → REJECTED → COA_SUBMITTED | COA_SUBMITTED 状态下退回（CoA 表 status→DRAFT），批次状态不变 |
| 服务项目报告退回 | COA_SUBMITTED → REPORT_PENDING | COA_SUBMITTED → REPORT_PENDING（批次状态直接回退） |

**实现**：
- BatchStatus 枚举删除 REJECTED
- CoaStatus 枚举保持 DRAFT/SUBMITTED/REJECTED/APPROVED
- 细胞产品的 CoA 退回：只更新 CoA 表 status 为 DRAFT，批次保持 COA_SUBMITTED
- 服务项目的报告退回：批次状态 COA_SUBMITTED → REPORT_PENDING

#### 4.2.2 服务项目鉴定（IDENTIFICATION）机制

**决策**：IDENTIFICATION 作为批次状态（宏观阶段），内部子步骤通过 ProductionTask 管理（微观执行）

- 批次进入 IDENTIFICATION = 标志"重编程完成，进入鉴定阶段"
- 批次离开 IDENTIFICATION = 标志"所有鉴定项目完成"
- 鉴定子步骤（整合检测、核型、多能性、STR 等）都是 ProductionTask
- 鉴定任务没有严格 sequenceNo 排序，可分配给不同人员
- 鉴定完成由操作员/主管手动触发（因判定"全部完成"的逻辑灵活）
- 基因编辑的两轮克隆挑选：IDENTIFICATION → IN_PRODUCTION → IDENTIFICATION（状态机回退）

#### 4.2.3 交接机制（HANDOVER）

**决策**：允许多次交接，HANDOVER 作为轻量过渡状态

```
IN_PRODUCTION ⇄ HANDOVER（可循环多次）
```

- 每次交接创建一个 HANDOVER 类型的 ProductionTask
- 记录交接方、接收方、交接说明
- 接收确认后才能进入下一个状态
- 不限制交接次数

#### 4.2.4 鉴定不通过处理

**决策**：由主管判断如何处理

```
IDENTIFICATION
├── REPORT_PENDING    → 鉴定全部通过
├── IN_PRODUCTION     → 返工（主管决定）
└── SCRAPPED          → 报废（主管决定）
```

- 多能性质控不通过 → 可能只是实验问题 → IDENTIFICATION → IN_PRODUCTION
- 核型异常 → 该克隆不可救 → IDENTIFICATION → SCRAPPED
- 基因型不纯 → 换子克隆 → IDENTIFICATION → IN_PRODUCTION

#### 4.2.5 服务项目报告+CoA 流程

**决策**：简化流程，跳过独立 QC 环节

```
REPORT_PENDING → COA_SUBMITTED → RELEASED
                     ↑
                     └── REPORT_PENDING（退回修改）
```

- 跳过 QC_PASS → COA_PENDING → COA_SUBMITTED 链路
- 到报告环节时理论上所有鉴定/质检已通过
- 批准后直接放行（RELEASED）

#### 4.2.6 试剂盒效力验证

**决策**：必须完成后才能放行

- 效力验证作为可选质检项，创建批次时配置
- 如果指令单要求做效力检测 → 必须等完成 → 才能放行
- 效力检测 15-30 天的等待是正常业务过程（QC_IN_PROGRESS 状态持续）

#### 4.2.7 状态机模板结构

**决策**：方案A — 按产品线分组

```typescript
const TRANSITION_TEMPLATES: Record<ProductLine, Record<string, TransitionRule[]>> {
  CELL_PRODUCT: { /* 独立完整流程 */ },
```

**理由**：三条产品线差异远大于共性，分组方案可读性、维护性、安全性均优于扁平混合方案。

#### 4.2.8 任务模板策略

**决策**：Phase 2 定义任务骨架 + Phase 3 实现表单

- Phase 2：定义每个状态转换时创建哪些任务（taskCode/taskName/sequenceNo）

## 8. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 状态机改造影响现有流程 | 细胞产品线当前可用的功能回退 | Phase 2 先确保现有流程不受影响，再逐步添加新模板 |

| 6+产品的表单开发量大 | Phase 3 周期过长 | 优先实现最关键的表单，次要表单用通用表单替代 |
| SQLite 不支持某些高级查询 | 多产品线联查性能问题 | 当前数据量小，SQLite 足够；后续可迁移 |
