## 📖 文档说明

### 文档目的
本文档旨在为 iPSC 实验室建立一套**完整、可执行的生产管理流程体系**，实现以下目标：

1. **流程标准化**：将隐性的工作经验转化为显性的操作规范，使新人可快速接手
2. **记录数字化**：解决当前文件夹管理导致的记录不及时、缺漏、人员流失后追溯困难的问题
3. **质量可追溯**：从成品到原料/种子的完整追溯链条
4. **系统建设蓝图**：为正在规划中的**生产管理系统**提供完整的功能需求基础

### 适用范围
- 服务生产组（建系、基因编辑、分化服务）
- 细胞产品组（现货iPSC细胞、现货分化细胞）
- 试剂盒生产组（培养基、分化试剂盒）
- QA 质量保证
- 仓管/发货

### 与现有体系的关系
本规划**不替代**已有的质量管理体系文件，而是在其框架下，对**生产执行层面的具体操作流程、记录模板、岗位协作接口**进行细化和补全。

## 第一部分：现状诊断与核心痛点

### 1.1 当前业务全景

实验室目前运营三条业务线：

| 业务线 | 模式 | 核心交付物 | 成熟度 |
|--------|------|-----------|--------|
| **服务项目** | 订单驱动（定制化） | 实物细胞 + 实验报告 | ⭐⭐⭐ 较成熟，有SOP |
| **现货细胞** | 库存驱动（预制备） | 冻存细胞株 / 分化细胞 | ⭐⭐ 流程有但记录混乱 |
| **试剂盒** | 产品驱动（批量生产） | 培养基、分化试剂盒 | ⭐ 正在建设中 |

### 1.2 当前组织结构

```
                    ┌─ 研发部（新方法开发 → 向生产转移）
                    │
                    │   ┌─ A：建系鉴定 + 部分分化

│              生产组（职责未完全区分，人员紧缺）

│   QA同事 × 几人（无独立QC团队）
│   仓管 × 1（库存 + 试剂配制 + 发货）
└──────────────────┘
```

### 1.4 核心痛点汇总

#### 🔴 P0 — 急需解决

| # | 痛点 | 影响 | 根因分析 |
|---|------|------|---------|

### 2.2 岗位职责详解

#### 2.2.1 生产主管

| 职责类别 | 具体内容 |
|---------|---------|
| **日常管理** | 统筹服务组和细胞产品组的任务分配与进度跟踪 |
| **技术指导** | 解决实验过程中的技术问题，审批关键操作 |
| **人员管理** | 新人培训安排、能力评估、绩效反馈 |
| **质量把关** | 审核CoA、批准批次放行、处理不合格品 |
| **跨部门协调** | 与研发部对接新技术转移，与QA对接质量问题，与仓管对接物料 |
| **持续改进** | 推动流程优化和效率提升 |

#### 2.2.2 服务生产组

| 职责 | 技能要求 | 特点 |
|------|---------|------|
| **重编程建系** | 细胞培养、重编程技术、克隆挑选 | 周期长（1-2个月），技术门槛高 |
| **建系鉴定** | 流式/IF检测、核型送检对接、STR、突变测序 | 建系的自然延伸，即"过程质检" |
| **基因编辑** | CRISPR/Cas9操作、单克隆筛选、阳性鉴定 | 编辑过程中同步完成鉴定 |
| **定制分化服务** | 多种分化方案执行 | 按客户需求定制 |

**关键规则**：
- 建系项目建议采用**交接制**：A完成初步建系和早代冻存 → B完成后续扩增和全套鉴定
- 这样既能保证专业性（每人专精），又能形成**互相检查**的机制
- 服务项目的完整SOP已存在，本规划重点关注**与服务结束→细胞产品开始之间的交接接口**

#### 2.2.3 细胞产品组

| 职责 | 具体内容 | 标准化程度 |
|------|---------|----------|
| **种子接收** | 从研发或服务组接收合格种子细胞，核对信息 | 🔴 需建标准 |
| **iPSC扩增冻存** | 复苏、传代扩增、按规格分装冻存 | ⭐⭐ 较高 |
| **现货分化生产** | 按既定方案执行分化（10-30天）、收获冻存 | ⭐ 中等 |
| **产品抽检/质检** | 每批必检：活率、形态、支原体；必要时：核型/标志物 | 🔴 需建标准 |
| **试剂盒功能检验** | 接受试剂盒组委托，用实际细胞做效力验证 | 🔴 需建标准 |

**关键特点**：此组工作**标准化程度高**，适合作为新人的入门培养区。薪资结构低于服务组，但有清晰的技能成长路径。

#### 2.2.4 QA（质量保证）

> 注意：目前没有独立的 QC（质量控制）团队，质检操作由生产人员执行，QA负责监督。

| 职责 | 具体内容 |
|------|---------|
| **文件体系管理** | SOP、表单、记录模板的受控管理 |
| **审核批准** | CoA审核、关键批次记录审核、偏差报告审核 |
| **偏差管理** | 组织偏差调查、CAPA跟踪 |
| **内部审计** | 定期检查各流程的合规性 |
| **供应商管理** | 关键原材料供应商的质量评估 |
| **变更控制** | 评估流程/方法变更的影响 |

#### 2.2.5 仓管

| 职责 | 具体内容 |
|------|---------|
| **物料管理** | 原料/包材入库、存储、出库、盘点 |
| **试剂配制** | 常用培养基/缓冲液的配制（正在建立表单） |
| **成品库存** | 成品入库登记、库位管理（配合细胞库存系统） |
| **发货管理** | 接收订单指令、打包、干冰/液氮安排、快递跟进 |
| **温湿度记录** | 冷链设备温度监控记录 |

### 2.3 能力矩阵与培训路径

#### 新人入职培训路径（以细胞产品组为例）

```
第1周: 安全培训 + GMP基础知识 + 实验室规章制度
        ↓ 通过考核
第2-3周: 基础操作训练（无菌技术、细胞复苏/传代/冻存/计数）
        ↓ 带教老师签核确认
第4-6周: 跟岗学习（跟随资深人员完成1-2个完整批次）
        ↓ 独立完成1个批次（主管监督下）
第7-8周: 正式上岗（独立操作，定期复核）
        ↓ 每6个月一次能力再评估
```

| 接口位置 | 交接内容 | 交接凭证 | 接收方确认 |

##### 与细胞产品的衔接
当客户提供的建系服务完成后，如果该细胞株后续要作为**现货细胞**销售时：
- 服务结束时额外冻存一支**种子级细胞**
- 填写《细胞株移交单》移交给细胞产品组
- 进入[现货细胞产品流程 §3.2]

| 7 | 突变测序 | Sanger测序 | 如适用 | 符合预期 | 基因编辑细胞必备 |

### 4.5 不合格品处理流程（基于现有质量体系）

> 以下为框架性描述，具体操作细节请参照质量体系中已建立的《不合格品处理程序》。

```
发现不合格（生产过程中或质检后发现）
       ↓
立即标识隔离（红色不合格标签 + 物理隔离）
       ↓
填写《不合格品报告》
       ↓
原因调查（根本原因分析）
       ├── 人：操作失误？培训不足？
       ├── 机：设备故障？
       ├── 料：原料问题？
       ├── 法：SOP问题？
       └── 环：环境/污染？
       ↓
制定纠正预防措施(CAPA)
       ↓
  ├── 方案A：返工处理（如果可行且经评估风险可控）
  ├── 方案B：让步接收（需特批，仅限 minor deviation）
  ├── 方案C：报废销毁
  └── 方案D：客户协商（已出货后发现问题）
       ↓
执行CAPA + 记录追踪关闭
```

### 6.1 系统定位与目标

**定位**：连接服务项目、细胞产品、试剂盒三大业务线的**统一生产管理平台**

**核心目标**：
1. **替代文件夹管理**：所有生产相关记录在线上完成，杜绝丢失和缺漏

4. **新人友好**：界面直观，操作提示清晰，降低培训成本
5. **追溯无忧**：一键查询任意批次的全生命周期信息

| 团队规模 | 小团队，不宜过度复杂 |
| 开发资源 | 看你们是否自主开发还是用现成平台 |
| 部署方式 | 建议私有部署（数据安全性） |
| 移动端需求 | 实验室操作可能需要平板/手机录入 |
| 预算 | 需要平衡功能和成本 |

#### 可选的技术路线

| 路线 | 适合场景 | 优点 | 缺点 |
|------|---------|------|------|
| **自研Web应用** | 有开发能力，需求高度定制 | 完全按需打造 | 开发和维护成本高 |
| **低代码平台（如宜搭/简道云）** | 快速上线，灵活调整 | 开发快，成本低 | 深度定制受限 |
| **专业MES/LIMS（轻量版）** | 标准化程度高的需求 | 功能成熟 | 价格贵，可能 overkill |
| **开源方案二次开发** | 有一定开发能力 | 成本可控，可定制 | 需要技术维护 |

> 💡 我的建议：考虑到你们的规模和"边补充边建设"的策略，可以先用**低代码平台快速搭建MVP（最小可行产品）**，跑通核心流程后再考虑是否迁移到自研或其他方案。
