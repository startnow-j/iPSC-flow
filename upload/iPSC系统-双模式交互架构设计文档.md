## 📋 文档说明

### 1.1 文档目的

本文档回答一个核心问题：

> **如果要实现「AI对话驱动」的生产管理体验，传统表单模式的系统在设计和开发时需要做哪些特殊准备，使得两种模式能够平滑共存？**

文档分两部分：
- **第一部分**：传统模式的特殊架构准备（技术侧）
- **第二部分**：AI对话模式的完整产品需求（产品侧）

### 1.2 核心论点

```
传统模式 ≠ AI模式的子集

传统模式是"数据操作的直接接口"
AI模式是"数据操作的智能代理"

两者操作同一份结构化数据，共用同一套业务规则。
关键在于：传统模式的架构必须为"可编程访问"做好准备，
而不是只做"人点击表单"的UI封装。
```

# 第一部分：传统模式的特殊架构准备

## 2. 问题分析：为什么需要提前准备

### 2.1 典型的"表单绑定"设计（❌ 不满足需求）

```
用户操作 → React表单组件 → handleSubmit() → 直接写数据库

问题：
1. 业务逻辑嵌在UI组件里，AI无法复用
2. 校验规则在前端，后端没有统一校验层
3. 状态流转在UI里硬编码，没有独立的状态机
4. 没有"意图 → 操作"的抽象层，AI不知道该调什么
```

### 2.2 正确的"分层"设计（✅ 满足需求）

```
┌─────────────────────────────────────────────────┐
│  表现层（Presentation）                          │
│  ├── 表单UI（人类直接操作）                       │
│  └── 对话AI（AI代理操作）                        │
│       ↓ 同一个接口                               │
│  意图层（Intent）                                │
│  └── 统一的意图识别 → 操作映射                    │
│       ↓                                         │
│  应用服务层（Application Service）               │
│  └── 纯业务逻辑，不关心是谁在调用                 │
│       ↓                                         │
│  领域层（Domain）                                │
│  └── 状态机、校验规则、业务约束                   │
│       ↓                                         │
│  数据层（Data）                                  │
│  └── 统一的数据模型和持久化                       │
└─────────────────────────────────────────────────┘
```

### 2.3 两层对比

| 维度 | 传统表单模式 | AI对话模式 | 共同依赖的底层能力 |
|------|------------|-----------|------------------|
| 输入方式 | 表单字段填写 | 自然语言 + 多模态 | — |
| 数据校验 | 前端实时校验 | AI提取后校验 | **统一的校验服务** |
| 状态流转 | 点击按钮触发 | AI判断后触发 | **统一的状态机** |
| 数据写入 | 表单提交 → API | 确认后 → 同一个API | **统一的应用服务** |
| 审计日志 | 提交即记录 | 对话+确认双重记录 | **统一的审计框架** |
| 错误处理 | 表单内联提示 | 对话中自然语言反馈 | **统一的错误码体系** |

## 3. 七项关键架构准备

### 3.1 【准备一】操作意图层（Intent Layer）

**为什么需要**：AI说"记录一下今天P4到P5传代"，系统需要知道这对应哪个具体操作。

**设计**：定义一套标准化的「操作意图」体系，表单模式和对话模式都映射到同一套意图。

```
意图定义示例：

意图：record_passage
  触发条件：用户汇报传代操作
  必要参数：
    - batch_id：批次ID
    - from_passage：传代前代次（如P4）
    - to_passage：传代后代次（如P5）
    - ratio：传代比例（如1:8）
    - density：细胞密度
    - observation：形态观察描述
  可选参数：
    - media_batch_no：培养基批号
    - photos：形态照片
    - operator：操作人
  关联数据写入：
    - production_task 表（更新当前任务进度）
    - production_record 表（新增传代记录行）
  关联业务规则：
    - 校验代次递增（P4→P5合法，P4→P3非法）
    - 校验传代比例在合理范围
    - 自动计算培养天数
```

**数据模型**：

```sql
CREATE TABLE operation_intent (
    id          BIGINT PRIMARY KEY,
    intent_code VARCHAR(64) NOT NULL COMMENT '意图编码',
    intent_name VARCHAR(128) NOT NULL COMMENT '意图名称',
    description TEXT COMMENT '意图描述',
    
    -- 参数定义（JSON Schema格式）
    params_schema JSON NOT NULL COMMENT '参数定义，JSON Schema',
    required_params JSON NOT NULL COMMENT '必填参数列表',
    
    -- 映射关系
    target_service VARCHAR(128) NOT NULL COMMENT '目标应用服务',
    target_method VARCHAR(128) NOT NULL COMMENT '目标方法',
    
    -- 上下文依赖
    requires_context JSON COMMENT '需要的上下文信息',
    -- 如: ["batch_id", "current_passage", "product_template"]
    
    -- 适用的产品线
    applicable_product_lines JSON COMMENT '适用产品线',
    -- 如: ["IPSC", "NPC", "CM"]
    
    status VARCHAR(16) DEFAULT 'ACTIVE',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
) COMMENT='操作意图定义表';
```

**关键点**：这个表是两种模式的"桥梁"。
- 表单模式的"提交"按钮 → 查找对应的 intent_code → 调用 target_service
- 对话模式的AI理解 → 提取参数 → 查找对应的 intent_code → 调用同一个 target_service

### 3.2 【准备二】服务器端统一校验服务

**为什么需要**：校验逻辑不能只放在前端表单里，AI调API时必须也能触发同样的校验。

**设计**：每一条校验规则都是一个独立的、可编程访问的服务。

```
校验服务设计原则：

1. 所有校验在后端执行（前端只做即时反馈，不作为最终校验）
2. 校验规则与产品模板绑定，随配置变化
3. 校验结果结构化返回，便于AI理解并反馈给用户

校验服务API：
POST /api/validation/validate
{
    "intent_code": "record_passage",
    "params": {
        "batch_id": 123,
        "from_passage": "P4",
        "to_passage": "P5",
        "ratio": "1:8",
        "density": 320000
    },
    "context": {
        "product_template_id": "tmpl-ipsc-001",
        "current_batch_status": "IN_PRODUCTION"
    }
}

响应：
{
    "valid": false,
    "errors": [
        {
            "field": "to_passage",
            "rule": "passage_increment",
            "message": "代次必须递增，当前为P4，目标P5与上一次传代P4相同",
            "severity": "error"
        }
    ],
    "warnings": [
        {
            "field": "density",
            "rule": "density_range",
            "message": "细胞密度3.2×10⁵偏高，正常范围为1-3×10⁵，请确认",
            "severity": "warning"
        }
    ]
}
```

**关键点**：
- 校验返回的是**结构化的错误信息**，AI可以翻译成自然语言告诉用户
- `severity` 分为 `error`（阻断提交）和 `warning`（提示但可继续）
- 校验规则来自产品模板配置，不是硬编码

### 3.3 【准备三】独立的状态机服务

**为什么需要**：批次状态的流转规则必须集中管理，表单按钮和AI操作都走同一套状态机。

**设计**：状态机作为独立的领域服务，不嵌入UI。

```
状态机定义（以iPSC细胞产品为例）：

states:
  NEW → IN_PRODUCTION → QC_PENDING → QC_IN_PROGRESS 
    → QC_PASS / QC_FAIL
    → COA_PENDING → COA_REVIEW → RELEASED → IN_STORAGE → SHIPPED
    
transitions:
  - { from: NEW, to: IN_PRODUCTION, trigger: start_production, roles: [PRODUCTION_TEAM] }
  - { from: IN_PRODUCTION, to: QC_PENDING, trigger: submit_production, roles: [PRODUCTION_TEAM] }
  - { from: QC_PENDING, to: QC_IN_PROGRESS, trigger: start_qc, roles: [QC_TEAM, QA] }
  - { from: QC_IN_PROGRESS, to: QC_PASS, trigger: pass_qc, roles: [QC_TEAM] }
  - { from: QC_IN_PROGRESS, to: QC_FAIL, trigger: fail_qc, roles: [QC_TEAM] }
  - { from: QC_PASS, to: COA_PENDING, trigger: generate_coa, roles: [SYSTEM] }
  - { from: COA_REVIEW, to: RELEASED, trigger: approve_coa, roles: [QA, SUPERVISOR] }
  - ...

状态机API：
POST /api/batch/{id}/transition
{
    "action": "submit_production",
    "operator": "张三",
    "reason": "传代培养完成，准备提交质检"
}

响应：
{
    "success": true,
    "previous_state": "IN_PRODUCTION",
    "new_state": "QC_PENDING",
    "next_available_actions": [
        { "action": "start_qc", "label": "开始质检", "allowed_roles": ["QC_TEAM", "QA"] }
    ]
}
```

**AI需要的额外能力**：

```sql
-- AI需要能查询：当前状态下可以做什么
GET /api/batch/{id}/available-actions

响应：
{
    "batch_id": 123,
    "current_state": "IN_PRODUCTION",
    "available_actions": [
        {
            "action": "record_passage",
            "label": "记录传代",
            "intent_code": "record_passage",
            "description": "录入一次传代操作记录"
        },
        {
            "action": "record_observation",
            "label": "记录观察",
            "intent_code": "record_observation",
            "description": "录入日常形态观察记录"
        },
        {
            "action": "submit_production",
            "label": "提交生产完成",
            "intent_code": "complete_production",
            "description": "标记生产完成，进入质检阶段"
        },
        {
            "action": "report_exception",
            "label": "上报异常",
            "intent_code": "report_exception",
            "description": "上报生产过程中的异常情况"
        }
    ]
}
```

**关键点**：AI通过这个API知道自己"能帮用户做什么"，不会凭空编造操作。

### 3.5 【准备五】结构化的操作记录与审计日志

**为什么需要**：表单提交和AI对话确认最终都要产生可审计的操作记录，格式需要统一。

**设计**：定义统一的审计事件模型。

```
审计事件模型：

无论是表单提交还是AI对话确认，都产生以下结构：

{
    // 事件标识
    "event_id": "evt-20260412-001",
    "event_type": "PRODUCTION_RECORD",
    "intent_code": "record_passage",
    
    // 操作信息
    "operator_id": "user-003",
    "operator_name": "张三",
    "operation_time": "2026-04-12T14:30:00",
    "input_mode": "AI_CONVERSATION",  // 或 "FORM_SUBMIT"
    
    // 目标对象
    "target_type": "BATCH",
    "target_id": "batch-123",
    "batch_no": "IPSC-260410-003-P5",
    
    // 操作数据（结构化）
    "data_before": { ... },   // 操作前的快照
    "data_after": {           // 操作后的数据
        "passage_from": "P4",
        "passage_to": "P5",
        "ratio": "1:8",
        "density": 320000,
        "morphology": "正常",
        "media_batch_no": "Media-20260405"
    },
    
    // AI模式特有的上下文
    "ai_context": {
        "conversation_id": "conv-456",
        "user_message": "今天P4到P5传代，比例1:8，密度3.2×10⁵，形态正常",
        "ai_extracted": { ... },     // AI提取的原始结果
        "user_confirmed": true,       // 用户是否确认
        "user_modifications": {},     // 用户修改了什么
        "model_version": "gpt-4o"     // 使用的AI模型
    },
    
    // 表单模式特有的上下文（两者选其一）
    "form_context": {
        "form_id": "form-ipsc-expansion-v1",
        "step": 3,
        "submission_duration_ms": 4500
    }
}
```

**关键点**：
- `input_mode` 字段区分来源，审计时可筛选
- AI模式下额外保存对话原文和AI提取结果，满足完整追溯
- 两种模式产生的 `data_after` 结构完全一致，下游报表/CoA不关心数据来源

### 3.6 【准备六】确认卡片的数据协议

**为什么需要**：AI提取的数据需要以可视化卡片形式让用户确认，这个卡片的数据结构需要在架构层面标准化。

**设计**：定义"确认卡片"的协议格式，前端根据协议渲染，不关心数据来自表单还是AI。

```
确认卡片协议（Confirmation Card Protocol）：

{
    "card_type": "CONFIRMATION",
    "intent_code": "record_passage",
    "title": "传代记录确认",
    "summary": "P4 → P5 传代，1:8比例",
    
    "data_groups": [
        {
            "group_name": "传代信息",
            "fields": [
                {
                    "code": "passage_from",
                    "label": "传代前",
                    "value": "P4",
                    "source": "ai_extracted",  // ai_extracted / user_input / auto_filled
                    "confidence": 0.95,       // AI提取置信度
                    "editable": false         // 用户可否修改
                },
                {
                    "code": "passage_to",
                    "label": "传代后",
                    "value": "P5",
                    "source": "ai_extracted",
                    "confidence": 0.95,
                    "editable": false
                },
                {
                    "code": "ratio",
                    "label": "传代比例",
                    "value": "1:8",
                    "source": "ai_extracted",
                    "confidence": 0.99,
                    "editable": true
                },
                {
                    "code": "density",
                    "label": "细胞密度",
                    "value": "3.2×10⁵ cells/cm²",
                    "source": "ai_extracted",
                    "confidence": 0.90,
                    "editable": true
                },
                {
                    "code": "media_batch_no",
                    "label": "培养基批号",
                    "value": "Media-20260405",
                    "source": "auto_filled",  // 系统自动填充
                    "editable": true,
                    "auto_fill_note": "沿用上次记录"
                }
            ]
        },
        {
            "group_name": "形态观察",
            "fields": [
                {
                    "code": "morphology",
                    "label": "形态判定",
                    "value": "正常",
                    "source": "ai_extracted",
                    "editable": true
                },
                {
                    "code": "photos",
                    "label": "形态照片",
                    "value": ["photo_001.jpg", "photo_002.jpg"],
                    "source": "user_attachment",
                    "editable": true
                }
            ]
        }
    ],
    
    "validation_result": {
        "valid": true,
        "warnings": [
            {
                "field": "density",
                "message": "细胞密度偏高，请确认数值是否正确"
            }
        ]
    },
    
    "actions": [
        { "type": "confirm", "label": "确认提交", "style": "primary" },
        { "type": "edit", "label": "修改", "style": "secondary" },
        { "type": "cancel", "label": "取消", "style": "ghost" }
    ]
}
```

**前端渲染逻辑**：

```
通用确认卡片渲染器：
  输入：Card Protocol JSON
  输出：可视化的确认界面

  渲染规则：
  - confidence >= 0.95：绿色标记，直接显示
  - confidence >= 0.80：黄色标记，提示"请核实"
  - confidence < 0.80：红色标记，高亮提示"请确认"
  - source = auto_filled：灰色小字标注"系统自动填充"
  - source = user_attachment：显示缩略图
  - editable = false：只读显示
  - editable = true：可点击修改
  
  表单模式和AI模式都使用同一个渲染器
```

            "action": "record_passage",
            "label": "记录传代",
            "expected_date": "2026-04-13",
            "hint": "根据上次传代记录，预计今天需要进行P4→P5传代"
        }
    ],

        "media_batch_no": "Media-20260405",
        "incubator": "CO2-02号",
        "operator": "张三"
    },
    
    "template_hints": {
        "current_step_fields": [ ... ],  // 当前步骤需要填写的字段（同3.4）
        "passage_history": ["P3→P4 1:6"],  // 历史传代记录

    }
}
```

**关键点**：这个API是AI的"记忆"，确保AI知道：
- 当前在哪个步骤
- 之前做过什么
- 默认值是什么（培养基批号、培养箱等）
- 下一步应该做什么

# 第二部分：AI对话模式 — 产品需求规格

## 5. 模块概述

### 5.1 模块定位

**AI对话工作台**是iPSC生产管理系统的可选交互模式，与表单模式并行存在。用户通过自然语言与系统交互，AI理解意图、提取结构化数据、生成确认卡片，用户确认后写入数据库。

### 5.2 核心用户价值

| 用户角色 | 痛点 | AI对话模式的解决方式 |
|---------|------|-------------------|
| 生产操作员 | 每天重复填大量表单，耗时且枯燥 | 自然语言30秒完成录入 |
| 生产操作员 | 戴手套不方便操作电脑/手机 | 语音输入 + 拍照直接发送 |
| 生产主管 | 需要逐个查看批次才能掌握进度 | 自然语言查询汇总 |
| QA | CoA审核需要逐项核对 | AI自动比对，高亮异常项 |

### 5.3 使用原则

| 原则 | 说明 |
|------|------|
| **确认优先** | AI提取的数据必须经用户确认才写入，不自动提交 |
| **数据等同** | 对话模式写入的数据与表单模式完全一致，下游无感知 |
| **可降级** | AI不可用时自动/手动切换到表单模式，不中断工作 |
| **可追溯** | 对话原文 + AI提取结果 + 用户确认，三层记录完整保存 |
| **渐进增强** | AI能力持续优化，但不影响已有功能的稳定性 |

### 6.2 模块二：AI对话引擎

#### 6.2.1 功能描述

后端AI服务，负责理解用户自然语言、提取结构化数据、调用业务API。

> **重要约束**：本模块运行在后端（服务器端），使用 z-ai-web-dev-sdk 调用LLM能力。前端不直接调用AI模型。

#### 6.2.2 技术架构

```
用户消息

#### 6.2.3 System Prompt 设计要点

```
System Prompt 核心内容：

1. 角色定义
   "你是iPSC生产管理系统的AI助手，帮助实验室生产人员
    通过自然语言录入生产记录、查询生产进度。"

2. 产品知识
   - 当前批次的产品类型和流程定义
   - 当前步骤需要收集哪些字段
   - 每个字段的含义和常见表述方式

3. 操作规则
   - 你可以执行的操作列表（来自 available-actions API）
   - 每个操作需要哪些参数
   - 参数的校验规则和正常范围

4. 上下文感知
   - 当前批次的历史操作记录
   - 默认值（培养基批号、操作人等）
   - 下一步应该做什么

5. 输出格式
   - 必须返回结构化的意图+参数
   - 必须标注每个提取值的置信度
   - 无法确定的信息标记为需要用户确认

6. 安全约束
   - 不要自动提交任何数据，必须等待用户确认
   - 不确定时主动询问，不要猜测
   - 涉及质量判定时保持客观，不代替人工判断
```

CREATE TABLE conversation (
    id          BIGINT PRIMARY KEY,
    session_id  VARCHAR(64) NOT NULL COMMENT '会话ID',

    INDEX idx_user (user_id)
) COMMENT='对话会话表';

    conversation_id BIGINT NOT NULL,
    role        ENUM('USER', 'AI', 'SYSTEM') NOT NULL,
    
    -- 消息内容
    content     TEXT COMMENT '消息文本内容',
    content_type ENUM('TEXT', 'IMAGE', 'FILE', 'CARD') DEFAULT 'TEXT',
    attachments JSON COMMENT '附件列表',
    
    -- AI消息的元数据
    ai_metadata JSON COMMENT 'AI消息的元数据',
    -- 包含：意图识别结果、参数提取结果、置信度等
    
    -- 关联的业务操作
    related_intent_code VARCHAR(64) COMMENT '关联的操作意图编码',
    related_action_id    BIGINT COMMENT '确认后产生的操作记录ID',
    
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_conversation (conversation_id)

### 6.4 模块四：语音与多模态输入

#### 6.4.1 功能描述

支持语音输入和图片输入，适应实验室实际操作场景。

#### 6.4.2 语音输入

| 项目 | 说明 |
|------|------|
| 触发方式 | 对话输入框旁的麦克风按钮 |
| 语音识别 | 后端 ASR 服务（z-ai-web-dev-sdk ASR能力） |
| 支持语言 | 中文（普通话）、英文 |
| 识别结果 | 转为文字后进入AI对话流程，与手动输入一致 |
| 适用场景 | 戴手套无法打字、边操作边记录 |
| 隐私 | 语音数据不持久化，识别完成后立即删除音频 |

#### 6.4.3 图片输入

| 项目 | 说明 |
|------|------|
| 触发方式 | 对话中直接粘贴/拖拽图片，或点击附件按钮上传 |
| 支持格式 | JPG, PNG, HEIC（手机拍照格式） |
| 图片处理 | 后端存储 + 生成缩略图 |
| AI处理 | 可选：调用 VLM 分析图片内容（如细胞形态判定） |
| 自动关联 | 图片自动关联到当前批次的当前步骤 |
| 适用场景 | 显微镜拍照后直接发送、上传检测图谱 |

#### 6.4.4 图片AI分析（进阶功能，P2）

```
用户发送显微镜照片后：

🤖 我看到这张照片中的iPSC细胞形态：
   · 集落呈圆形/椭圆形，边缘清晰 ✅
   · 细胞核大、核仁明显 ✅
   · 未观察到明显分化区域 ✅
   
   AI判定：形态正常
   
   是否同意这个判定？
   [同意] [不同意，我的判断是___]
```

## 7. 两种模式的交互矩阵

### 7.1 按场景的推荐模式

| 场景 | 推荐模式 | 备选模式 | 说明 |
|------|---------|---------|------|
| 日常传代记录 | 🗣️ 对话 | 📋 表单 | 高频重复，对话效率高 |
| 细胞形态观察+照片 | 🗣️ 对话 | 📋 表单 | 拍照直接发送最便捷 |
| 培养基更换记录 | 🗣️ 对话 | 📋 表单 | 信息简单，一句话搞定 |
| 质检结果录入 | 🗣️ 对话 | 📋 表单 | 数据格式标准，AI提取准确 |
| 异常上报 | 🗣️ 对话 | 📋 表单 | 随时发生，快速记录 |
| 新建批次 | 📋 表单 | 🗣️ 对话 | 需要选择多个配置项，表单更直观 |
| 批次详情查看 | 📋 结构化页 | 🗣️ 对话 | 需要看到完整的时间线和数据 |
| CoA审核 | 📋 结构化审批 | — | 需要逐项核对，不适合对话 |
| 追溯查询 | 🗣️ 对话 | 📋 表单搜索 | 自然语言查询更直观 |
| 报表统计 | 📋 仪表盘 | 🗣️ 对话 | 可视化图表更适合结构化展示 |
| 产品/模板配置 | 📋 配置器 | — | 复杂配置操作，仅管理员使用 |
| 种子库管理 | 📋 表单 | — | 数据精确性要求高 |

### 7.2 模式切换规则

```
┌────────────────────────────────────────────────────┐
│                   模式切换决策树                     │
├────────────────────────────────────────────────────┤
│                                                    │
│  用户打开一个批次                                   │
│       │                                            │
│       ├─ 检查：用户是否有"默认模式"设置？            │
│       │   ├─ 是 → 使用默认模式                     │
│       │   └─ 否 → 检查当前步骤推荐模式              │
│       │                                            │
│       ├─ 检查：AI服务是否可用？                     │
│       │   ├─ 可用 → 使用推荐模式                   │
│       │   └─ 不可用 → 强制使用表单模式              │
│       │       └─ 显示提示："AI服务暂时不可用，       │
│       │          当前使用表单模式"                    │
│       │                                            │
│       └─ 用户可随时手动切换模式                     │
│                                                    │
│  模式切换时数据保持：                                │
│  · 对话模式已确认的数据 → 表单中显示为已填写         │
│  · 表单模式已填写的数据 → 对话中作为上下文           │
│  · 两种模式写入同一张表，数据格式一致                │
│                                                    │
└────────────────────────────────────────────────────┘
```

## 9. 非功能性需求

### 9.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| AI响应时间 | ≤ 3秒（P95） | 从用户发送到AI回复首字节 |
| 确认卡片渲染 | ≤ 500ms | 收到卡片数据到渲染完成 |
| 数据提交 | ≤ 1秒 | 用户确认到数据入库 |
| 表单模式响应 | ≤ 200ms | 不受AI性能影响 |
| 降级切换 | ≤ 1秒 | 检测到AI不可用到切换完成 |

### 9.2 可靠性要求

| 指标 | 要求 |
|------|------|
| AI可用性 | ≥ 99%（允许每月约7小时不可用） |
| 数据一致性 | 两种模式写入的数据结构100%一致 |
| 降级覆盖 | AI不可用时所有功能可在表单模式下完成 |
| 数据不丢失 | 对话中未确认的数据在本地暂存，不丢失 |

### 9.3 安全与合规

| 项目 | 要求 |
|------|------|
| 数据隔离 | 不同用户的对话数据严格隔离 |
| AI数据安全 | 对话内容不用于训练AI模型（选择合规的LLM服务） |
| 审计完整性 | 对话原文 + AI提取结果 + 用户确认 三层记录不可篡改 |
| 电子签名 | 关键操作（质检判定、CoA审批）仍需独立电子签名，对话确认不替代签名 |
| LLM调用安全 | API Key 存储在服务端环境变量，不暴露给前端 |

## 10. 实施建议

### 10.1 开发顺序

```
Phase 1（MVP）：先做传统模式，但架构到位
  ┌──────────────────────────────────────────────┐
  │ ① 独立状态机服务                              │
  │ ② 统一校验服务                                │
  │ ③ 统一审计日志                                │
  │ ④ 表单模式完整功能（iPSC细胞产品线）           │
  │ ⑤ 表单模式完整功能（试剂盒产品线）             │
  │ ⑥ 操作意图定义（表单提交走意图层）             │
  └──────────────────────────────────────────────┘
  ↓ 基础设施就绪
Phase 2：叠加AI对话层
  ┌──────────────────────────────────────────────┐
  │ ⑦ 上下文服务                                  │
  │ ⑧ 确认卡片协议 + 通用渲染器                   │
  │ ⑨ 对话引擎（后端）                            │
  │ ⑩ 对话工作台（前端）                          │
  │ ⑪ 对话记录与审计                              │
  │ ⑫ 模式切换机制                                │
  └──────────────────────────────────────────────┘
  ↓ 
Phase 3：增强体验
  ┌──────────────────────────────────────────────┐
  │ ⑬ 语音输入                                   │
  │ ⑭ 图片AI分析（VLM）                          │
  │ ⑮ 智能查询与汇总                              │
  │ ⑯ AI主动提醒（基于上下文预判下一步）           │
  └──────────────────────────────────────────────┘
```

### 10.2 关键风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM幻觉（编造数据） | 数据准确性受损 | 确认卡片 + 结构化校验双重保障 |
| LLM响应慢 | 影响工作效率 | 超时自动降级到表单模式 |
| AI提取错误 | 用户需要反复修改 | 置信度标注 + 上下文学习用户习惯 |
| 对话歧义 | AI理解错误意图 | 多轮确认、不猜测 |
| 审计合规 | 新模式可能不满足GMP要求 | 三层记录 + 保留传统表单模式 |

## 11. 总结

### 11.1 核心设计理念

```
对话是入口，不是替代品。
结构化数据是根基，不会因为入口变化而动摇。
两种模式操作同一份数据，共用一套规则。
确认环节不可省略，审计追溯不可打折。
AI增强效率，但不改变责任归属。
```

### 11.2 对原PRD的影响

| 原PRD模块 | 影响 | 说明 |
|-----------|------|------|
| 模块二：批次管理 | 无变化 | 数据模型不变 |
| 模块三：eBPR | 接口调整 | 表单提交改为走意图层 → 应用服务 |
| 模块四：质检管理 | 接口调整 | 同上 |
| 模块五：CoA管理 | 无变化 | CoA从结构化数据生成，不关心数据来源 |
| 模块一：工作台 | 重大变化 | 增加对话工作台作为主要入口 |
| 新增模块 | AI对话引擎 | 后端服务，调用LLM |
