# iPSC-Flow 项目文档管理规范

> **生效日期**：2026-04-14
> **最后更新**：2026-06-18
> **适用对象**：所有参与项目的开发人员（含 AI 辅助开发）

---

## 1. 目录结构

```
iPSC-flow/
├── worklog.md              # AI 工作日志（每次任务完成后追加）
├── GIT_WORKFLOW.md         # Git 工作流规范（提交、分支、安全规则）
├── PROJECT_DOCS.md         # 本文件 — 文档管理规范
│
├── docs/                   # 📌 项目核心文档（纳入 Git 版本管理）
│   ├── requirements/       #    需求文档
│   │   └── PRD-v2.0-MVP.md
│   ├── architecture/       #    架构设计
│   │   └── 双模式交互架构设计文档.md
│   └── plans/              #    开发计划与方案
│       ├── 开发计划-融合版-v1.0.docx
│       └── 多产品线扩展计划-v1.0.md
│
├── archive/                # 归档（本地保留，不进 Git）
│   └── [已被替代或过期的文档]
│
└── upload/                 # 用户上传（临时文件，不进 Git）
    └── [用户上传的参考文档]
```

### 目录职责

| 目录 | 是否进 Git | 说明 |
|------|:----------:|------|
| `docs/` | ✅ | 当前有效的项目文档，是项目的知识核心 |
| `archive/` | ❌ | 已被替代、过期或仅供后续参考的文档 |
| `upload/` | ❌ | 用户上传的原始参考文件 |
| 根目录 `*.md` | ✅ | 项目级规范文件（worklog、GIT_WORKFLOW、PROJECT_DOCS） |

## 2. 文档分类规则

### 2.1 何时归档（移入 archive/）

满足以下任一条件，文档应移入 `archive/`：

| 条件 | 示例 |
|------|------|
| 文档有更新版本替代 | PRD v1.0 → PRD v2.0 发布后，v1.0 归档 |
| 文档中的方案被推翻 | 某技术方案经评审后放弃 |
| 文档仅作为后续阶段参考 | Phase 3 才用到的设计文档，当前不需要 |
| 文档内容已过时 | 基于旧版需求编写的流程方案 |

### 2.2 归档命名规则

归档时在文件名中标注状态，便于识别：

```
原文：PRD.md
归档：PRD-v1.0-已废弃.md

原文：AI模板生成模块设计文档.md
归档：AI模板生成模块设计文档-Phase3参考.md

原文：开发计划.md
归档：开发计划-旧版-已废弃.md
```

后缀选项：`-已废弃`、`-Phase3参考`、`-早期素材`、`-后续参考`

### 2.3 何时删除

满足以下条件可**直接删除**（不归档）：

- AI 生成的临时分析文件
- 重复的内容
- 无任何参考价值的草稿

## 4. worklog.md 管理规则

`worklog.md` 是 AI 辅助开发的连续工作记录，必须遵循以下格式：

### 4.1 条目格式

每次任务完成后，在文件**末尾追加**一个条目：

```markdown
---
Task ID: <递增编号>
Agent: <执行者名称>
Task: <简要任务描述>

Work Log:
- <具体步骤1>
- <具体步骤2>
- <具体步骤3>

Stage Summary:
- <关键结果 / 重要决策 / 产出物>
```

### 4.2 维护规则

| 规则 | 说明 |
|------|------|
| 只追加，不修改已有条目 | 保持审计完整性 |
| 单个条目不超过 10 行 Work Log | 太长的拆分为子任务 |
| Stage Summary 必须有 | 方便后续 Agent 快速了解上下文 |
| 文件超过 200 行时清理 | 删除超过 30 天的已完成条目，保留摘要 |

## 5. 定期清理规则

AI 在以下时机**必须检查并清理**文档：

### 5.1 每次对话开始时

1. 读取 `worklog.md` 了解上次做了什么
2. 检查 `docs/` 目录是否有新增的过时文档需要归档
3. 检查 `archive/` 目录是否有超过 60 天且无价值的文件需要删除

### 5.2 每次重大决策后

当出现以下情况时，立即评估文档状态：

- PRD 更新 → 旧版 PRD 归档
- 架构方案调整 → 更新架构文档
- 开发计划修改 → 旧版计划归档
- 新增设计文档 → 放入对应 docs/ 子目录

### 5.3 文档版本管理

当更新文档时：
1. 更新文档内的版本号和日期
2. 在文档顶部更新"修订记录"章节
3. 如果是重大变更（如 PRD 从 v1 升级到 v2），先归档旧版再创建新版

## 6. docs/ 子目录扩展规则

随着项目推进，`docs/` 目录可按需扩展：

```
docs/
├── requirements/       # 需求：PRD、用户故事、验收标准
├── architecture/       # 架构：系统设计、接口规范、数据模型
├── plans/              # 计划：开发计划、迭代计划、风险评估
├── api/                # API：接口文档（开发过程中自动生成）
├── guides/             # 指南：部署手册、用户操作手册
└── decisions/          # 决策记录：ADR（Architecture Decision Records）
```

新增子目录时需同步更新本文件的第3节"当前有效文档清单"。
